## Goal

ถามผู้ใช้ก่อนทำงานที่มีความเสี่ยงสูงเพื่อลดความผิดพลาด

## Scope

ใช้สำหรับงานที่กระทบไฟล์จำนวนมาก, เปลี่ยน architecture, ลบไฟล์, หรือเปลี่ยน behavior หลัก

## Execute

### 1. Assess Risk

ประเมินความเสี่ยงของงาน:

1. ตรวจสอบจำนวนไฟล์ที่กระทบ
2. ตรวจสอบว่าเปลี่ยน architecture หรือ pattern หลักหรือไม่
3. ตรวจสอบว่ามีการลบไฟล์หรือไม่
4. ตรวจสอบว่าเปลี่ยน behavior ของระบบหรือไม่

### 2. Ask User

ถามผู้ใช้ก่อนทำงานที่เสี่ยงสูง:

1. สรุปสิ่งที่จะทำเป็น bullet points สั้นๆ
2. ระบุไฟล์ที่จะกระทบ
3. ระบุความเสี่ยงที่อาจเกิดขึ้น
4. ถามผู้ใช้เพื่อยืนยันก่อนดำเนินการ

### 3. Proceed Or Abort

1. ถ้าผู้ใช้ยืนยัน ให้ทำตามแผน
2. ถ้าผู้ใช้ปฏิเสธ ให้หยุดและรอคำสั่งใหม่

## Rules

### 1. High-Risk Actions

ต้องถามผู้ใช้ก่อนทำ:

- ลบไฟล์หรือโฟลเดอร์
- เปลี่ยน architecture หรือ pattern หลัก
- แก้ไขไฟล์มากกว่า 10 ไฟล์พร้อมกัน
- เปลี่ยน behavior ของระบบหลัก
- ติดตั้งหรืออัปเดต dependencies หลัก
- ทำงานที่ไม่ reversible

### 2. Low-Risk Actions

ไม่ต้องถาม ทำได้เลย:

- แก้ไขไฟล์ 1-3 ไฟล์
- เพิ่มไฟล์ใหม่
- แก้ typo หรือ formatting
- รัน lint, typecheck, test
- อ่านไฟล์หรือ search code

### 3. Communication

- ถามสั้นกระชับ ไม่ยาวเกินจำเป็น
- ระบุเฉพาะสิ่งที่จะทำและความเสี่ยง
- ไม่ถามซ้ำถ้าผู้ใช้ตอบแล้ว

## Expected Outcome

- ลดความผิดพลาดจากการทำงานที่เสี่ยงสูง
- ผู้ใช้มีส่วนร่วมในการตัดสินใจ
- ไม่ทำงานที่ผู้ใช้ไม่ต้องการ
