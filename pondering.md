## Goal

ใช้เวลาครุ่นคิด (ponder) อย่างลึกซึ้งและรอบคอบ เพื่อทบทวนมุมมองหลายด้านก่อนตัดสินใจหรือดำเนินการใดๆ

## Scope

ใช้สำหรับงานที่ต้องการการครุ่นคิด ทบทวน หรือพิจารณาอย่างรอบคอบ ไม่ใช่การวิเคราะห์เชิงระบบ (ใช้ `/deep-thinking`) และไม่ใช่การวางแผนงาน (ใช้ `/plan`)

## Execute

### 1. Pause And Reflect

หยุดและทบทวนก่อนดำเนินการ:

1. ระบุสิ่งที่กำลังจะตัดสินใจหรือทำ
2. ถามตัวเองว่า "ควรทำเลยไหม หรือควรคิดก่อน?"
3. ถ้าเป็น low-risk และชัดเจน ให้ทำเลยโดยไม่ต้อง ponder
4. ถ้ามีความเสี่ยง ซับซ้อน หรือไม่แน่ใจ ให้ดำเนินการ ponder

### 2. Consider Multiple Perspectives

พิจารณาจากหลายมุมมอง:

1. มุมมองของผู้ใช้ — ผู้ใช้ต้องการอะไรจริงๆ
2. มุมมองของระบบ — impact ต่อ codebase, performance, security
3. มุมมองของอนาคต — ผลกระทบระยะยาว, maintainability
4. มุมมองของ trade-offs — อะไรได้และอะไรเสียจากแต่ละทางเลือก

### 3. Weigh Implications

ชั่งน้ำหนักผลกระทบของแต่ละทางเลือก:

1. ระบุผลกระทบที่คาดว่าจะเกิดขึ้น (immediate และ downstream)
2. ประเมินความรุนแรงของผลกระทบ (low, medium, high)
3. ระบุความเสี่ยงที่อาจเกิดขึ้น
4. พิจารณาว่าสามารถย้อนกลับได้หรือไม่ (reversible vs irreversible)
5. ถ้า irreversible ให้คิดให้รอบคอบเป็นพิเศษ

### 4. Surface Uncertainties

ระบุสิ่งที่ไม่แน่ใจ:

1. ระบุ unknowns ทั้งหมด
2. แยกระหว่าง known unknowns (รู้ว่าไม่รู้) และ unknown unknowns (ไม่รู้ว่าไม่รู้)
3. ประเมินว่า unknowns สำคัญพอที่ต้องหาข้อมูลเพิ่มหรือไม่
4. ถ้าจำเป็น ให้ใช้ `/ask-me` เพื่อถามผู้ใช้ หรือ `/deep-research` เพื่อค้นหาข้อมูล

### 5. Reach Conclusion

สรุปผลการครุ่นคิด:

1. สรุปแนวทางที่เหมาะสมที่สุดจากการทบทวน
2. ระบุเหตุผลที่เลือกแนวทางนี้
3. ระบุข้อจำกัดและความเสี่ยงที่ยอมรับได้
4. ถ้ายังไม่ชัดเจน ให้ใช้ `/ask-me` เพื่อขอคำตัดสินใจจากผู้ใช้
5. ถ้าชัดเจนแล้ว ให้ดำเนินการต่อ

## Rules

### 1. When To Ponder

ครุ่นคิดเมื่อ:

- การตัดสินใจมีผลกระทบสูงหรือ irreversible
- มีหลายทางเลือกที่ดีพอๆ กัน
- ไม่แน่ใจว่าผู้ใช้ต้องการอะไรจริงๆ
- การกระทำอาจมี side effects ที่ไม่คาดคิด
- ต้องเลือกระหว่าง speed และ quality

### 2. When Not To Ponder

ไม่ครุ่นคิดเมื่อ:

- เป็น low-risk action ที่ทำได้เลย
- คำตอบชัดเจนจาก context หรือ memory
- ผู้ใช้ระบุชัดเจนว่าต้องการอะไร
- เป็น reversible change ที่แก้ไขง่าย
- การคิดนานกว่าการทำจะเสียเวลามากกว่า

### 3. Pondering vs Deep Thinking

- `/deep-thinking` = วิเคราะห์เชิงระบบ แบ่งปัญหา สร้างทางเลือก ตรวจสอบ assumptions
- `/pondering` = ทบทวนมุมมอง ชั่งน้ำหนัก พิจารณาผลกระทบ ระบุความไม่แน่ใจ
- ใช้ `/deep-thinking` เมื่อต้องวิเคราะห์ปัญหาเชิงโครงสร้าง
- ใช้ `/pondering` เมื่อต้องทบทวนและพิจารณาก่อนตัดสินใจ

### 4. Quality Of Pondering

คุณภาพของการครุ่นคิดต้อง:

- ครอบคลุมหลายมุมมอง ไม่ยึดติดมุมมองเดียว
- พิจารณาทั้งระยะสั้นและระยะยาว
- ระบุ assumptions และ uncertainties อย่างตรงไปตรงมา
- ไม่ overthink จนไม่ได้ลงมือทำ
- สรุปได้ชัดเจนว่าจะทำอะไรต่อ

### 5. Time Budget

- ถ้าเป็น decision เล็ก ใช้เวลา ponder ไม่เกิน 1 นาที
- ถ้าเป็น decision กลาง ใช้เวลา ponder ไม่เกิน 3 นาที
- ถ้าเป็น decision ใหญ่ ใช้เวลา ponder ตามความจำเป็น แต่ต้องสรุปได้
- ถ้า ponder นานเกินไป ให้ใช้ `/ask-me` เพื่อให้ผู้ใช้ตัดสินใจแทน

### 6. Integration With Other Workflows

เชื่อมโยงกับ workflows อื่น:

- ทำ `/deep-thinking` หลัง ponder ถ้าต้องวิเคราะห์เชิงระบบ
- ทำ `/ask-me` ถ้า ponder แล้วยังไม่ชัดเจน
- ทำ `/plan` หลัง ponder ถ้าต้องวางแผนงาน
- ทำ `/suggest-next-action` หลัง ponder ถ้าต้องรู้ว่าควรทำอะไรต่อ

## Expected Outcome

- การตัดสินใจที่รอบคอบและครอบคลุม
- ลดความเสี่ยงจากการกระทำโดยไม่ทบทวน
- เข้าใจผลกระทบทั้งระยะสั้นและระยะยาว
- ระบุความไม่แน่ใจก่อนดำเนินการ
- สร้างความมั่นใจก่อนตัดสินใจ
