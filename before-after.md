## Goal

อธิบายแผนการทำงานก่อนดำเนินการ และแสดง before-after ให้เหมาะสมโดยไม่แก้ไขจริง

## Execute

### 1. Analyze Request

1. วิเคราะห์สิ่งที่ผู้ใช้ต้องการ
2. ระบุสิ่งที่จะเปลี่ยนแปลง
3. ระบุผลกระทบที่คาดว่าจะเกิด

### 2. Explain Plan

1. ตอบว่าจะทำอะไร ทำอย่างไร
2. อธิบายขั้นตอนการทำงาน
3. ระบุไฟล์ที่จะเกี่ยวข้อง

### 3. Show Before

1. แสดงสภาพปัจจุบัน (before)
2. แสดงปัญหาหรือสิ่งที่ต้องปรับปรุง
3. อธิบายเหตุผลที่ต้องเปลี่ยน

### 4. Show After

1. แสดงสภาพหลังเปลี่ยน (after)
2. แสดงผลลัพธ์ที่คาดหวัง
3. อธิบายประโยชน์ที่จะได้รับ

### 5. Confirm With User

1. ถามผู้ใช้ยืนยันก่อนดำเนินการ
2. รอการอนุมัติจากผู้ใช้
3. ดำเนินการเมื่อได้รับอนุมัติ

## Rules

### 1. Explain First

ต้องอธิบายก่อนดำเนินการ:

- บอกว่าจะทำอะไร
- บอกว่าจะทำอย่างไร
- บอกว่าจะกระทบอะไร
- รอการยืนยันจากผู้ใช้

### 2. Before After Format

รูปแบบการแสดง before-after:

```markdown
### Before
[สภาพปัจจุบัน]

### After
[สภาพหลังเปลี่ยน]
```

### 3. No Actual Changes

ยังไม่ต้องแก้ไขจริง:

- แสดงแผนการเท่านั้น
- ไม่แก้ไขไฟล์จริง
- รอการยืนยันจากผู้ใช้ก่อน

### 4. Clear Comparison

ต้องแสดงความแตกต่างอย่างชัดเจน:

- ใช้ table เปรียบเทียบถ้าเหมาะสม
- ใช้ code block แสดง before/after code
- ใช้ bullet points แสดงการเปลี่ยนแปลง

### 5. Use Workflows

ใช้ workflows ที่เกี่ยวข้อง:

- ทำ `/plan` เพื่อวางแผน
- ทำ `/analyze-project` เพื่อวิเคราะห์
- ทำ `/follow-write-workflows` เพื่อสร้าง workflow

## Expected Outcome

- ผู้ใช้เข้าใจแผนการทำงาน
- แสดง before-after อย่างชัดเจน
- ไม่มีการแก้ไขจริงจนกว่าจะได้รับอนุมัติ
- ผู้ใช้ยืนยันก่อนดำเนินการต่อ
