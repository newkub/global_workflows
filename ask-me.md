## Goal

ถามผู้ใช้คำถามพร้อมตัวเลือกที่กำหนดไว้และมีคำแนะนำ (recommended) เพื่อขอคำยืนยัน ตัดสินใจเลือกทางเลือก หรือขอข้อมูลเพิ่มเติม

## Scope

ใช้สำหรับการถามคำถามทั่วไปที่ต้องการคำตอบแบบเลือกตัวเลือก ไม่ใช่เก็บ requirements (ใช้ `/ask-requirement`) และไม่ใช่สัมภาษณ์ preferences (ใช้ `/understand-me`)

ใช้ `ask_user_question` tool จาก Windsurf system prompt **เท่านั้น** ห้ามใช้วิธีอื่นในการถามผู้ใช้

## Execute

### 1. Identify Question Type

ระบุประเภทคำถามที่จะถาม

1. วิเคราะห์ว่าต้องการคำตอบแบบใด: `confirmation`, `choice`, `multiple-choice`, หรือ `clarification`
2. ถ้าเป็น `confirmation` ให้ตัวเลือกแบบ yes/no หรือ proceed/abort
3. ถ้าเป็น `choice` ให้ตัวเลือก 2-4 ตัวเลือกที่เป็นทางเลือกเดียว
4. ถ้าเป็น `multiple-choice` ให้ตัวเลือกที่เลือกได้หลายตัว
5. ถ้าเป็น `clarification` ให้ตัวเลือกที่ช่วยให้ผู้ใช้ระบุสิ่งที่ต้องการชัดเจนขึ้น

### 2. Prepare Options With Recommendation

เตรียมตัวเลือกสำหรับคำถามพร้อมระบุตัวเลือกที่แนะนำ

1. กำหนดตัวเลือก 2-4 ตัวเลือกตามประเภทคำถาม
2. แต่ละตัวเลือกต้องมี `label` สั้นกระชับ และ `description` อธิบายรายละเอียด
3. ระบุตัวเลือกที่แนะนำโดยเพิ่ม `(recommended)` ต่อท้าย `label` และอธิบายเหตุผลใน `description`
4. ไม่ใส่ตัวเลือก "other" เพราะผู้ใช้สามารถตอบเองได้เสมอ
5. ตัวเลือกต้องครอบคลุมทุกกรณีที่เป็นไปได้
6. เรียงลำดับตัวเลือกโดยวางตัวเลือกที่แนะนำไว้ลำดับแรก

### 3. Ask Question

ถามผู้ใช้โดยใช้ `ask_user_question` tool

1. เขียน `question` ให้ชัดเจน กระชับ และมี context เพียงพอ
2. ส่ง `options` ที่เตรียมไว้พร้อม `label` และ `description`
3. ตั้ง `allowMultiple` เป็น `true` ถ้าเป็น `multiple-choice`
4. ถ้าต้องถามหลายคำถาม ให้ถามทีละคำถาม ไม่ถามพร้อมกัน

### 4. Process Response

ประมวลผลคำตอบจากผู้ใช้

1. นำคำตอบไปดำเนินการตามที่ผู้ใช้เลือก
2. ถ้าผู้ใช้ให้คำตอบเอง (custom response) ให้วิเคราะห์และปรับแผนตามคำตอบ
3. ถ้าผู้ใช้เลือกตัวเลือกที่ต้องการข้อมูลเพิ่ม ให้ถามคำถามตามมา
4. ไม่ถามซ้ำคำถามที่ผู้ใช้ตอบแล้ว

## Rules

### 1. When To Ask

ถามผู้ใช้เมื่อ:

- มีทางเลือกที่ต้องตัดสินใจและไม่ชัดเจนว่าผู้ใช้ต้องการอะไร
- ต้องการยืนยันทิศทางก่อนดำเนินการ
- ข้อมูลไม่เพียงพอที่จะดำเนินการได้ถูกต้อง
- มีหลายวิธีที่ทำได้และผู้ใช้ควรเลือก

### 2. When Not To Ask

ไม่ถามผู้ใช้เมื่อ:

- คำตอบชัดเจนจาก context หรือจาก memory
- เป็นการเก็บ requirements (ใช้ `/ask-requirement`)
- เป็นการสัมภาษณ์ preferences (ใช้ `/understand-me`)
- เป็น low-risk action ที่ทำได้เลย

### 3. Question Design

- คำถามสั้นกระชับ มี context เพียงพอ
- ใช้ภาษาที่ผู้ใช้เข้าใจ หลีกเลี่ยง jargon
- ตัวเลือกครอบคลุมทุกกรณีที่เป็นไปได้
- `label` สั้นไม่เกิน 3-5 คำ
- `description` อธิบายรายละเอียดและเหตุผลของตัวเลือก
- ไม่ใส่ตัวเลือก "other" หรือ "none"
- เรียงลำดับโดยตัวเลือกที่แนะนำอยู่ลำดับแรก

### 4. Recommendation

- ทุกคำถามต้องมีตัวเลือกที่แนะนำอย่างน้อย 1 ตัวเลือก
- ระบุ `(recommended)` ต่อท้าย `label` ของตัวเลือกที่แนะนำ
- อธิบายเหตุผลใน `description` ว่าทำไมแนะนำตัวเลือกนี้
- คำแนะนำต้องอ้างอิงจาก context, best practices หรือ project conventions
- ถ้ามีหลายตัวเลือกที่ดีพอๆ กัน ให้เลือกตัวที่ง่ายที่สุดเป็น recommended

### 5. Interaction Patterns

- ถามทีละคำถาม ไม่ถามพร้อมกัน
- ถ้าผู้ใช้ตอบเอง ให้ปรับแผนตามคำตอบ
- ไม่ถามซ้ำคำถามที่ตอบแล้ว
- ถ้าคำตอบนำไปสู่คำถามใหม่ ให้ถามตามมา
- หลังได้คำตอบ ให้ดำเนินการทันทีไม่ชะลอ

## Expected Outcome

- ผู้ใช้ตัดสินใจได้ง่ายจากตัวเลือกที่ชัดเจนและมีคำแนะนำ
- ลดการคาดเดาของ AI
- การทำงานสอดคล้องกับความต้องการของผู้ใช้
- ลดเวลาการสื่อสารโดยไม่จำเป็น
